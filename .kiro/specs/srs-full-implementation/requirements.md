# Requirements Document

## Introduction

This document covers the full SRS compliance audit and gap-closure requirements for the PharmaLink Pro Pharmacy Wholesale & Importer Website. The system is built with React (frontend), Node.js/Express (backend), and PostgreSQL. Significant implementation already exists; this specification targets the delta between the current state and 100% SRS compliance, plus the additional features listed at the end of the SRS.

The twelve functional gaps identified through code audit are organized into requirements below. Each requirement maps to one or more SRS sections and describes precisely what must be built or fixed.

---

## Glossary

- **System**: The PharmaLink Pro web application (frontend + backend combined).
- **Navbar**: The top navigation component (`Navbar.jsx`) rendered on all public-facing pages.
- **RFQ_Wizard**: The 4-step request-for-quotation form (`RFQ.jsx`).
- **RFQ_Store**: The Zustand client-side state store (`rfqStore.js`) that holds wizard state.
- **RFQ_Success_Page**: The dedicated post-submission confirmation page (`RFQSuccess.jsx`) at route `/rfq/success/:rfqNumber`.
- **Customer_Portal**: The authenticated customer area under `/portal/*`.
- **Customer_Dashboard**: The customer RFQ history page (`CustomerDashboard.jsx`).
- **Customer_RFQ_Detail**: The per-RFQ detail page for customers (`CustomerRFQDetail.jsx`).
- **Admin_Panel**: The authenticated admin area under `/admin/*`.
- **Compare_Page**: The product comparison page (`Compare.jsx`).
- **Products_Page**: The product catalog page (`Products.jsx`).
- **Register_Page**: The customer registration page (`Register.jsx`).
- **i18n_System**: The react-i18next localization system initialized in `i18n.js` with `en.json` and `am.json` translation files.
- **Translation_Key**: A dot-notation key in `en.json` / `am.json` used by the `useTranslation()` hook.
- **Language_Toggle**: A UI control in the Navbar that switches the active locale between English and Amharic.
- **Track_RFQ_Page**: The public RFQ status lookup page (`TrackRFQ.jsx`) at route `/track`.
- **Home_Contact_Section**: A section at the bottom of the Home page containing address, phone, email, map embed, and a quick contact form.
- **SEO_Meta**: HTML `<meta>` tags (title, description, Open Graph) and JSON-LD schema markup injected per page.
- **Dosage_Form**: A product attribute describing the physical form of a medicine (tablet, capsule, injection, syrup, etc.).
- **Country_of_Origin**: A product attribute indicating the country where the product was manufactured.
- **Per_Product_Note**: A free-text note field attached to a single line item in the RFQ, stored in `rfq_items.notes`.
- **Quotation_PDF**: The formal quotation document generated by the admin and sent to the customer.
- **RFQ_PDF**: The submitted RFQ document (not the quotation) that the customer can download immediately after submission.
- **Admin**: An authenticated user with `role = 'admin'`.
- **Customer**: An authenticated user with `role = 'customer'`.
- **Visitor**: An unauthenticated user browsing the public site.

### Requirement 1: Multi-Language Support — Full Application Coverage

**User Story:** As a Visitor or Customer, I want to toggle the website language between English and Amharic, so that I can use the site in my preferred language across all pages.

#### Acceptance Criteria

1. THE Navbar SHALL display a Language_Toggle control that is visible on all screen sizes (desktop and mobile menu).
2. WHEN a user activates the Language_Toggle, THE i18n_System SHALL switch the active locale to the selected language and persist the selection to `localStorage`.
3. WHEN the active locale is changed, THE System SHALL re-render all page text using the corresponding Translation_Keys from `en.json` or `am.json` without a full page reload.
4. THE Home page SHALL use `useTranslation()` hooks for all user-visible text strings instead of hardcoded English literals.
5. THE Products_Page SHALL use `useTranslation()` hooks for all user-visible text strings instead of hardcoded English literals.
6. THE RFQ_Wizard SHALL use `useTranslation()` hooks for all user-visible text strings instead of hardcoded English literals.
7. THE Compare_Page SHALL use `useTranslation()` hooks for all user-visible text strings instead of hardcoded English literals.
8. THE About page SHALL use `useTranslation()` hooks for all user-visible text strings instead of hardcoded English literals.
9. THE Services page SHALL use `useTranslation()` hooks for all user-visible text strings instead of hardcoded English literals.
10. THE Contact page SHALL use `useTranslation()` hooks for all user-visible text strings instead of hardcoded English literals.
11. THE Navbar SHALL use `useTranslation()` hooks for all navigation labels and action button text.
12. THE Footer SHALL use `useTranslation()` hooks for all user-visible text strings.
13. IF a Translation_Key is missing in the active locale, THEN THE i18n_System SHALL fall back to the English value without displaying a raw key string.

### Requirement 2: Product Comparison — Up to 4 Products with Full SRS Fields

**User Story:** As a Visitor or Customer, I want to compare up to 4 products side by side including dosage form and country of origin, so that I can make informed procurement decisions.

#### Acceptance Criteria

1. THE Compare_Page SHALL allow a user to select up to 4 products for simultaneous side-by-side comparison (current limit of 3 must be raised to 4).
2. WHEN a user attempts to add a fifth product to the comparison, THE Compare_Page SHALL disable the add action for unselected products and display a message indicating the 4-product limit has been reached.
3. THE Compare_Page comparison table SHALL include a "Dosage Form" row displaying the `dosage_form` attribute of each selected product.
4. THE Compare_Page comparison table SHALL include a "Country of Origin" row displaying the `country_of_origin` attribute of each selected product.
5. WHEN a product does not have a `dosage_form` or `country_of_origin` value, THE Compare_Page SHALL display "—" in the corresponding cell.
6. THE products database table SHALL include `dosage_form VARCHAR(100)` and `country_of_origin VARCHAR(100)` columns.
7. THE Admin_Panel product management form SHALL include input fields for Dosage Form and Country of Origin.
8. THE backend `/products/:id` endpoint SHALL return `dosage_form` and `country_of_origin` fields in the product response object.

### Requirement 3: RFQ Wizard Step 2 — Per-Product Notes UI

**User Story:** As a Customer, I want to add a note to each individual product in my RFQ, so that I can communicate specific requirements (e.g., preferred strength, packaging preference) per line item.

#### Acceptance Criteria

1. THE RFQ_Wizard Step 2 selected-products panel SHALL display a text input field labeled "Note" for each product line item in the selected list.
2. WHEN a Customer types in the per-product note input, THE RFQ_Store SHALL update the `notes` field of the corresponding line item via `updateProduct()`.
3. THE per-product note input SHALL accept up to 500 characters.
4. THE RFQ_Wizard Step 4 review panel SHALL display each product's note beneath the product name if the note is non-empty.
5. WHEN an RFQ is submitted, THE System SHALL persist each line item's `notes` value to the `rfq_items.notes` column in the database.
6. THE Admin_Panel RFQ detail view SHALL display per-product notes beneath each product name in the items list.
7. THE Customer_RFQ_Detail page SHALL display per-product notes beneath each product name in the items list.

### Requirement 4: RFQ Submission — Dedicated Success Page with PDF Download

**User Story:** As a Customer, I want to see a dedicated success page after submitting my RFQ that shows my RFQ number and lets me download a PDF copy, so that I have immediate confirmation and a record of my submission.

#### Acceptance Criteria

1. WHEN an RFQ is submitted successfully, THE RFQ_Wizard SHALL navigate the Customer to the RFQ_Success_Page at `/rfq/success/:rfqNumber` instead of redirecting to `/portal`.
2. THE RFQ_Success_Page SHALL display the unique RFQ number prominently (e.g., `RFQ-2026-0001`).
3. THE RFQ_Success_Page SHALL display a "Download RFQ PDF" button that, when clicked, downloads the submitted RFQ as a PDF file.
4. WHEN the "Download RFQ PDF" button is clicked, THE System SHALL call the backend endpoint to generate and stream the RFQ PDF to the browser.
5. THE RFQ_Success_Page SHALL display a "What Happens Next" section describing the review and quotation process.
6. THE RFQ_Success_Page SHALL display a link to the Track_RFQ_Page pre-populated with the submitted RFQ number.
7. THE RFQ_Success_Page SHALL display a link to the Customer_Dashboard.
8. IF the RFQ number in the URL does not correspond to an existing RFQ, THEN THE RFQ_Success_Page SHALL display an error message and a link to the Customer_Dashboard.

### Requirement 5: Customer Portal — Prominent Quotation PDF Download

**User Story:** As a Customer, I want to easily download my formal quotation PDF from both the dashboard and the RFQ detail page, so that I can review pricing and terms without searching for the download option.

#### Acceptance Criteria

1. THE Customer_Dashboard SHALL display a "Download Quotation" button for every RFQ with status `QUOTATION_SENT`.
2. THE Customer_RFQ_Detail page SHALL display a prominently styled "Download Quotation PDF" button when the RFQ status is `QUOTATION_SENT`.
3. THE Customer_RFQ_Detail page SHALL display a "Download RFQ Copy" button for all RFQ statuses, allowing the customer to download the original submitted RFQ as a PDF.
4. WHEN a Customer clicks "Download Quotation PDF", THE System SHALL call `GET /customer/rfqs/:id/pdf` with the customer's JWT and stream the quotation PDF to the browser.
5. WHEN a Customer clicks "Download RFQ Copy", THE System SHALL call the appropriate backend endpoint and stream the submitted RFQ PDF to the browser.
6. IF the quotation PDF is not yet available (status is not `QUOTATION_SENT`), THEN THE Customer_RFQ_Detail page SHALL display an informational message explaining that the quotation will be available once the admin sends it.

### Requirement 6: SEO — Product Meta Tags and Schema Markup

**User Story:** As a site owner, I want each page to have proper meta tags and product pages to have schema markup, so that search engines can index the site correctly and display rich results.

#### Acceptance Criteria

1. THE System SHALL inject a unique `<title>` and `<meta name="description">` tag for each of the following pages: Home, Products, About, Services, Contact, RFQ, and Track RFQ.
2. THE Products_Page SHALL inject Open Graph meta tags (`og:title`, `og:description`, `og:url`, `og:image`) into the document `<head>`.
3. THE Home page SHALL inject Open Graph meta tags into the document `<head>`.
4. THE Products_Page SHALL inject a JSON-LD `ItemList` schema block listing the displayed products when the product list is loaded.
5. WHEN a product detail view or product card is rendered, THE System SHALL include the product name, brand, and category in the page's structured data.
6. THE System SHALL use canonical `<link rel="canonical">` tags on all public pages to prevent duplicate content indexing.
7. THE existing `sitemap.xml` SHALL be updated to include URLs for the About, Services, Contact, Products, and Track RFQ pages with `<lastmod>` dates.

### Requirement 7: Home Page — Contact Section with Map and Quick Form

**User Story:** As a Visitor, I want to see contact information, a map, and a quick contact form directly on the Home page, so that I can reach the company without navigating to a separate Contact page.

#### Acceptance Criteria

1. THE Home page SHALL include a Home_Contact_Section as the final content section before the footer.
2. THE Home_Contact_Section SHALL display the company's physical address, phone number, and email address.
3. THE Home_Contact_Section SHALL include an embedded map (iframe or map component) showing the company's location.
4. THE Home_Contact_Section SHALL include a quick contact form with fields: First Name, Last Name, Email, and Message.
5. WHEN a Visitor submits the quick contact form with valid data, THE System SHALL send the message to the backend contact endpoint and display a success confirmation message.
6. IF the quick contact form is submitted with an invalid or missing email address, THEN THE System SHALL display a validation error on the email field without submitting the form.
7. IF the quick contact form submission fails due to a backend error, THEN THE System SHALL display an error message and retain the form data so the user can retry.
8. THE Home_Contact_Section SHALL be fully responsive on desktop, tablet, and mobile viewports.
9. THE Home_Contact_Section text SHALL use Translation_Keys so it is covered by the i18n_System.

### Requirement 8: Customer Registration — Full Profile Fields

**User Story:** As a new Customer, I want to provide my company name, business type, phone, country, and city during registration, so that the RFQ wizard can auto-fill Step 1 without me re-entering the same information.

#### Acceptance Criteria

1. THE Register_Page SHALL include input fields for: Full Name, Company Name, Business Type (dropdown), Email, Phone, Country, City, Password, and Confirm Password.
2. THE Register_Page form SHALL validate that Company Name, Business Type, Phone, Country, and City are non-empty before submission.
3. WHEN a Customer submits the registration form with valid data, THE System SHALL send all profile fields to the backend `POST /auth/register` endpoint.
4. THE backend `POST /auth/register` endpoint SHALL accept and persist `company_name`, `business_type`, `phone`, `country`, and `city` fields to the `users` table.
5. THE backend `POST /auth/register` endpoint SHALL return the full user profile (including `companyName`, `businessType`, `phone`, `country`, `city`) in the JWT response payload.
6. WHEN a registered Customer opens the RFQ_Wizard, THE RFQ_Wizard Step 1 SHALL auto-fill all customer info fields from the authenticated user's profile stored in the auth store.
7. IF a Customer's profile is complete (all RFQ Step 1 fields are populated), THEN THE RFQ_Wizard SHALL automatically advance to Step 2 on load.

### Requirement 9: Track RFQ — Prominent Navigation Access

**User Story:** As a Visitor or Customer, I want to easily find the RFQ tracking feature from the main navigation and the Home page, so that I can check my RFQ status without knowing the direct URL.

#### Acceptance Criteria

1. THE Navbar SHALL include a "Track RFQ" navigation link that routes to `/track`.
2. THE Navbar "Track RFQ" link SHALL be visible in both the desktop navigation bar and the mobile menu.
3. THE Home page SHALL include a visible link or button to the Track_RFQ_Page in at least one prominent location (e.g., the CTA section or the hero area).
4. THE RFQ_Success_Page SHALL include a direct link to the Track_RFQ_Page pre-populated with the submitted RFQ number.
5. THE Customer_Dashboard success banner (shown after RFQ submission) SHALL include a "Track RFQ" link that routes to `/track`.

### Requirement 10: Admin Content Management — Product Categories Section

**User Story:** As an Admin, I want to manage the product categories displayed on the Home page through the content management panel, so that I can update category names, icons, and descriptions without code changes.

#### Acceptance Criteria

1. THE Admin_Panel content management page SHALL include a "Categories" tab or section for managing the product category cards shown on the Home page.
2. WHEN an Admin edits a category entry (name, icon, color, description), THE System SHALL persist the changes to the `site_content` table under the `home_categories` section key.
3. WHEN the Home page loads, THE System SHALL fetch the categories content from the `site_content` table and render the category cards using the stored data.
4. IF no categories content exists in the database, THEN THE Home page SHALL fall back to the hardcoded default `CATEGORIES` array.
5. THE Admin_Panel categories editor SHALL allow the Admin to reorder category entries.
6. THE Admin_Panel categories editor SHALL allow the Admin to add new category entries and remove existing ones.

### Requirement 11: Live Chat — Visitor Access and Admin Notifications

**User Story:** As a Visitor or Customer, I want to use a live chat widget to communicate with support staff in real time, so that I can get immediate help with procurement questions without needing to log in first.

**Note:** The polling-based live chat widget (`LiveChat.jsx`) and admin console (`AdminChat.jsx`) are already implemented. This requirement covers the gaps: the chat widget must be accessible to unauthenticated Visitors, and the admin must receive a notification badge for new messages.

#### Acceptance Criteria

1. THE LiveChat widget SHALL be rendered and accessible to unauthenticated Visitors on all public pages.
2. WHEN an unauthenticated Visitor opens the chat widget, THE System SHALL assign a guest session identifier and allow the Visitor to send messages without requiring login.
3. WHILE a chat session is active, THE System SHALL poll for new messages at an interval of no more than 5 seconds.
4. WHEN the Admin_Panel receives a new chat message from any session, THE System SHALL display a notification badge on the Admin chat navigation item indicating the count of unread messages.
5. THE LiveChat widget SHALL display a connection status indicator showing whether the support agent is online or offline.
6. IF the backend chat endpoint is unreachable, THEN THE LiveChat widget SHALL display an error state and allow the user to retry without losing their typed message.

### Requirement 12: Non-Functional — Performance, Security, and Responsiveness

**User Story:** As a site owner, I want the application to meet the performance, security, and responsiveness standards defined in the SRS, so that users have a fast, secure, and accessible experience on all devices.

#### Acceptance Criteria

1. THE System SHALL serve all public pages with a Time to First Contentful Paint of less than 3 seconds on a standard broadband connection.
2. WHEN an RFQ is submitted, THE System SHALL complete the full submission flow (form POST, database write, email dispatch) within 5 seconds under normal load.
3. THE System SHALL enforce HTTPS for all client-server communication in the production environment.
4. WHEN a file is uploaded as an RFQ attachment, THE System SHALL validate that the file's MIME type is one of `application/pdf`, `image/jpeg`, or `image/png` and that the file size does not exceed 10 MB.
5. IF an uploaded file fails MIME type or size validation, THEN THE System SHALL reject the upload and return a descriptive error message to the client.
6. THE System SHALL hash all user passwords using bcrypt with a cost factor of at least 12 before storing them in the database.
7. THE System SHALL use JWT-based authentication with tokens that expire after no more than 7 days.
8. THE System SHALL render all pages correctly on viewport widths of 320px (mobile), 768px (tablet), and 1280px (desktop).
9. THE System SHALL use semantic HTML elements and ARIA attributes on all interactive components to support screen reader accessibility.
10. WHEN a user's JWT has expired, THE System SHALL redirect the user to the login page and preserve the originally requested URL as a `redirect` query parameter.
